## 작업 과정에서 어려웠던 점 & 배운 점
### 추측하지 말고 한 번 더 확인하자
팀원과 협업하는 과정에서 두 차례 커뮤니케이션 오해가 발생했다.

데일리 스크럼에서 업무 분담을 진행한 결과, 나를 포함해 두 명이서 FE 디렉터리 구조를 결정하기로 하였다.

각자 생각하고 있는 방안를 GitHub Discussion에 업로드하고, 서로의 구조에서 좋은 부분들을 추출하여 통합본을 만드는 방향으로 진행해보자는 의견으로 좁혀졌다.

그 과정에서 나는 2개의 안을 올리게 되었는데, 팀원 분께서는 1안에 대한 코멘트를 달아주셨다.

그래서 코멘트에 답변드리고 1안을 수정하고 있었는데, '본인이 생각한 구조와 2안을 절충하여 통합본을 완성했다'며 확인해 달라는 요청이 들어왔다.

나는 당연히 내가 1안을 수정하면, 수정된 1안과 팀원의 구조에서의 장점들을 추출하여 통합본을 완성할 줄 알았는데, 전혀 다른 방식으로 진행이 된 것이었다. 

지금까지 내가 만나왔던 사람들은 보통 디벨롭시키고 싶은 방향에 코멘트를 달아주셨기 때문에, '이 사람도 당연히 그렇지 않을까'하고 넘겨 짚었던 것이 미스 커뮤니케이션의 원인이 되었다고 생각한다.

또한 오후에 작업하는 과정에서도, 팀원 분께서 하고 계시는 작업을 잘못 이해하는 일이 있었다. 전체 작업 프로세스가 A → B → C라고 할 때, 나는 그 분이 A → B 작업을 수행하고 계신 줄 알았는데, 중간에 확인해 보니 B → C 작업을 하고 계셨던 것이다.

이 경험들을 통해 '내가 이해했다고 바로 작업에 착수하는 것'이 아니라, '내가 상대와 동일하게 이해했는지 한 번 검증하는 과정을 거치는 것'이 좋겠다고 생각하게 되었다.

1분 안팎의 간단한 확인만으로도 시간과 에너지 낭비를 막을 수 있을 뿐 아니라, 상대방을 더욱 믿고 작업에 임할 수 있게 되기 때문이다.

# 2. 종합 회고
## 잘한 점
### 원활한 분업
- 이번 주 그룹 회고 내역을 보면, 공통적으로 나왔던 내용이 '아직 분업이 제대로 이루어지고 있지 않다'였다.
- 이 점을 고려해, 당일 데일리 스크럼에서는 업무를 크게 3개(FE, 이슈 할당, 컨벤션)로 나눠서 각각 2명, 2명, 1명에게 할당되도록 했는데, 결과적으로 모든 업무를 성공적으로 마칠 수 있었다.
- 모든 작업에 5명 전원이 붙는 것은 비효율적일 수 있다. 물론 전원이 참여해야 하는 업무도 있겠지만, 그렇지 않은 경우에는 앞으로도 적절히 분업해 나갈 수 있도록 해야겠다.

## 아쉬웠던 점
### 팀 작업 간 적절한 휴식을 취하지 못했다
- 개인 작업을 할 때는 원하는 타이밍에 쉴 수 있지만, 팀 작업을 진행하게 되면 쉴 타이밍을 잡지 못하게 되는 것 같다.
- 그라운드 룰로도 휴식 시간을 지정했었지만, 제대로 휴식 시간이 지켜지지 않아 작업 능률이 떨어진 점이 아쉬웠다.

## 개선할 점
### 의식적으로 휴식 시간을 챙기자
- 제대로 휴식이 이루어지지 못하면 작업 능률이 떨어질 뿐만 아니라, 그라운드 룰이 갖는 의미 또한 퇴색된다.
- 작업 흐름이 너무 끊기는 경우가 아니라면, 의식적으로 휴식 시간을 챙기려고 노력해 보자.

# 3. 학습 내용
## AWS EC2
### Instance Type
- EC2 인스턴스 타입의 이름은 해당 가상 서버의 하드웨어 특성과 성능 사양을 나타내는 체계적인 명명 규칙을 따른다. 
- `t4g.micro`를 예시로 각 항목의 의미를 분해하면 다음과 같다.
- **t (인스턴스 패밀리):** 인스턴스의 주요 용도와 성능 특성을 나타낸다.
  - t: 기본 수준의 성능을 제공하며 필요 시 CPU 성능을 버스팅(가속)할 수 있는 범용 인스턴스다.
  - m: 컴퓨팅, 메모리, 네트워크 리소스가 균형 있게 배분된 범용 인스턴스다.
  - c: 고성능 프로세서가 필요한 컴퓨팅 집약적 워크로드(예: 머신러닝 추론, 고성능 웹 서버)에 적합한 컴퓨팅 최적화 인스턴스다.
  - r: 메모리 집약적인 애플리케이션(예: 고성능 인메모리 캐시, 실시간 빅데이터 분석)에 적합한 메모리 최적화 인스턴스다.
- **4 (세대):** 
  - 하드웨어 아키텍처의 세대를 의미한다. 
  - 숫자가 높을수록 최신 세대의 하드웨어가 적용되어 성능과 비용 효율성이 높다.
- **g (추가 속성/프로세서 타입):** 탑재된 칩셋의 종류나 특수 기능을 명시한다.
  - g: AWS가 자체 설계한 ARM 기반의 Graviton 프로세서를 사용함을 의미한다.
  - a: AMD 프로세서를 의미한다. 
  - i: Intel 프로세서를 의미한다.
- **micro (인스턴스 크기):** 
  - 할당되는 vCPU 및 메모리의 용량을 나타낸다. 
  - `nano`, `micro`, `small`, `medium`, `large`, `xlarge` 순으로 리소스 용량이 배수로 증가한다.

## AWS Storage Service
### EBS (Elastic Block Store)
- EC2 인스턴스에 직접 연결하여 사용하는 블록(Block) 스토리지다. 
- 물리적인 서버에 장착하는 내장 SSD 또는 HDD와 동일한 방식으로 작동한다. 
- 데이터를 일정한 크기의 블록으로 나누어 저장하며, 단일 EC2 인스턴스에 마운트하여 운영체제(OS)를 설치하거나 파일 시스템을 포맷하여 사용한다.

### S3 (Simple Storage Service)
- 네트워크(HTTP/HTTPS)를 통해 API 호출로 접근하는 객체(Object) 스토리지로, 특정 서버에 종속되지 않는 독립적인 서비스다. 
- 데이터를 파일 계층 구조가 아닌 개별 객체 단위로 저장하며, 각 객체는 데이터 자체, 메타데이터, 그리고 고유한 식별자(URL)로 구성된다. 
- 저장 용량에 사실상 제한이 없다.

### 선택 가이드
EBS와 S3는 데이터를 처리하는 방식과 접근 구조가 완전히 다르기 때문에, 애플리케이션의 아키텍처 요구사항에 따라 명확히 구분하여 사용해야 한다.

#### EBS 사용이 적합한 상황 
- EBS는 단일 서버 내에서 빠르고 빈번한 읽기/쓰기 작업이 발생하고, 데이터의 일관성과 저지연(Low-Latency)이 필수적인 경우에 사용한다.
- **운영체제(OS) 부팅 디스크:** EC2 인스턴스를 실행하기 위한 루트 볼륨은 반드시 EBS로 구성해야 한다.
- **관계형 데이터베이스(RDBMS):** MySQL, PostgreSQL 등 트랜잭션 처리가 중요하고 디스크 I/O 성능이 애플리케이션 성능에 직결되는 데이터베이스의 저장소로 사용한다.
- **엔터프라이즈 애플리케이션:** 서버 내부의 파일 시스템에 직접 접근하여 **파일을 지속적으로 수정하고 덮어써야 하는 애플리케이션 워크로드에 적합**하다.

#### S3 사용이 적합한 상황
- S3는 데이터의 수정이 빈번하지 않으며, 여러 대의 서버나 외부 클라이언트가 동시에 접근해야 하는 대규모 비정형 데이터를 저장할 때 사용한다.
- **정적 웹 콘텐츠 미디어 저장:** 사용자가 업로드한 프로필 이미지, 동영상 파일, 문서 파일 등을 저장하고 서빙하는 데 사용한다. 여러 대의 EC2(WAS)가 동일한 파일에 동시 접근해야 하는 분산 환경에 필수적이다.
- **백업 및 아카이빙:** 데이터베이스의 주기적인 스냅샷 백업, 서버의 로그 파일, 규정 준수를 위해 장기 보관해야 하는 대용량 데이터를 저장할 때 저렴한 비용으로 활용한다.
- **데이터 레이크 (Data Lake):** 빅데이터 분석 및 머신러닝 모델 학습을 위해 다량의 원시(Raw) 데이터를 수집하고 보관하는 중앙 저장소 역할을 한다.

### 종합 비교 요약
| 구분 | EBS | S3 |
| --- | --- | --- |
| **스토리지 유형** | 블록(Block) 스토리지 | 객체(Object) 스토리지 |
| **접근 방식** | 단일 EC2에 마운트하여 파일 시스템으로 접근 | 인터넷(API, URL)을 통해 다수의 시스템에서 접근 |
| **성능 (I/O)** | 응답 속도가 매우 빠름 (지연 시간 최소화) | 처리량(Throughput)은 높으나 응답 지연이 존재 |
| **데이터 수정** | 파일 내 특정 바이트/블록만 수정 가능 | 파일 일부 수정 불가 (전체 객체를 덮어써야 함) |
| **저장 용량** | 볼륨 생성 시 용량 지정 (최대 제한 있음) | 용량 제한 없음 (사용한 만큼만 비용 지불) |

## AWS Serverless
### Serverless Computing
- 서버리스(Serverless)는 단어의 의미와 달리 서버가 존재하지 않는다는 것이 아니다. 
- 물리적 또는 가상의 서버 인프라에 대한 프로비저닝, 유지 보수, 용량 확장 등의 관리 작업을 클라우드 제공업체(AWS)가 전적으로 처리하는 클라우드 컴퓨팅 실행 모델을 의미한다.
- 개발자는 운영체제 패치, 서버 스케일링, 로드 밸런싱 등을 신경 쓸 필요 없이 오직 애플리케이션의 핵심 비즈니스 로직(코드)과 데이터 관리에만 집중할 수 있다. 
- 또한, 서버를 켜두는 시간에 비례하여 비용을 지불하는 기존 방식과 달리, 실제 코드가 실행되거나 자원이 소비된 만큼만 비용을 지불하는 종량제(Pay-as-you-go) 방식을 채택하고 있다.

### Lambda
- 서버를 프로비저닝하거나 관리하지 않고도 코드를 실행할 수 있게 해주는 이벤트 기반의 서버리스 컴퓨팅 서비스(FaaS, Function as a Service)다.
- **인프라 관리 제로:** EC2와 달리 운영체제 환경을 설정할 필요가 없다. Python, Java, Node.js 등으로 작성된 코드를 업로드하기만 하면 즉시 실행 가능한 상태가 된다.
- **자동 스케일링:** 요청이 들어올 때마다 AWS가 독립적인 실행 환경을 즉각적으로 생성하여 코드를 처리한다. 초당 1개의 요청이든 10,000개의 요청이든 트래픽 규모에 맞춰 자동으로 병렬 처리 단위가 확장된다.
- **밀리초 단위 과금:** 코드가 실행되지 않을 때는 과금되지 않으며, 코드가 실행된 시간(밀리초 단위)과 할당된 메모리 크기에 비례하여 요금이 청구된다.

### DynamoDB
- 어떤 규모에서든 10ms 미만의 일관된 응답 지연 시간을 제공하는 완전 관리형 서버리스 NoSQL 데이터베이스다.
- **자동 용량 관리:** 데이터베이스 서버의 인스턴스 크기(CPU, RAM)를 미리 결정할 필요가 없다. 트래픽 증감에 따라 읽기/쓰기 용량을 자동으로 확장 및 축소(Auto-scaling)한다.
- **고가용성 내장:** 데이터가 여러 가용 영역(AZ)에 걸쳐 자동으로 복제되므로, 별도의 이중화 클러스터링을 구성하지 않아도 데이터 손실을 방지하고 높은 가용성을 보장한다.
- **온디맨드 과금:** 용량을 미리 예약하지 않고 애플리케이션에서 수행한 데이터 읽기 및 쓰기 요청 횟수에 대해서만 과금하는 방식을 선택할 수 있다.

### S3는 왜 서버리스일까?
- **스토리지 서버 프로비저닝 불필요:** 사용자는 데이터를 저장하기 위해 디스크 공간이 얼마나 필요할지 예측하고 디스크 볼륨을 할당(EBS 방식)할 필요가 없다.
- **무한한 자동 스케일링:** 저장하는 데이터의 양에 맞춰 스토리지 공간이 자동으로 제한 없이 확장된다.
- **서버 인프라의 추상화:** 사용자는 API나 웹 콘솔을 통해 객체를 업로드하고 다운로드할 뿐, 그 이면에 있는 스토리지 서버 네트워크, 디스크 드라이브의 상태, 물리적 데이터 복제 과정을 전혀 관리하지 않는다.
- **사용량 기반 과금:** 미리 디스크 크기를 구매하는 것이 아니라, 실제로 S3 공간에 저장된 데이터의 용량(GB)과 네트워크 전송량에 대해서만 비용을 지불한다.

## 서버리스 아키텍처 동작 흐름
- 항상 서버가 실행된 상태로 트래픽을 대기하는 Spring Boot 기반의 EC2 아키텍처와 달리, 서버리스 환경에서는 클라이언트의 요청이 발생할 때마다 각 서비스가 이벤트 기반으로 동작하며 유기적으로 연결된다.
- 이러한 구조를 채택하면 개발자는 네트워크 라우팅 테이블이나 인스턴스 사양, 디스크 마운트를 신경 쓸 필요 없이 각 컴포넌트의 기능 구현과 연동 설정에만 집중할 수 있다.
- 프로젝트 적용 여부와는 무관하게, 동작 흐름을 한번 살펴보자.

### 1. 정적 콘텐츠 제공 (웹 프론트엔드)
- S3 기반 정적 웹 호스팅: 사용자가 클라이언트(브라우저)를 통해 서비스 URL로 접속하면, S3에 미리 저장해 둔 정적 리소스(HTML, CSS, JavaScript 파일 및 이미지)가 클라이언트 측으로 다운로드된다.
- 이 단계에서 S3는 데이터를 단순히 보관하는 것을 넘어, 무한히 확장 가능한 정적 웹 서버의 역할을 수행한다. 
- 갑자기 대규모 트래픽이 발생하더라도 별도의 스케일링 설정 없이 S3 자체적으로 부하를 처리한다.

### 2. API 요청 및 라우팅 (API Gateway)
- **엔드포인트 제공 및 트래픽 수신:** 브라우저에 로드된 클라이언트 애플리케이션이 데이터를 읽거나 쓰기 위해 REST API 요청(예: `POST /api/users`)을 전송하면, 이 요청은 가장 먼저 API Gateway로 인입된다.
- **요청 검증 및 트리거:** 
  - API Gateway는 인입된 HTTP 메서드와 URL 경로를 확인하고 인가(Authorization) 작업을 수행한다.
  - 유효한 요청으로 판별되면, 해당 API 엔드포인트와 매핑되어 있는 특정 Lambda 함수를 호출(Trigger)하여 트래픽을 넘긴다.

### 3. 비즈니스 로직 실행 (Lambda)
- **코드의 즉각적인 실행 (컴퓨팅):** API Gateway의 호출 이벤트가 발생하면, AWS는 즉시 해당 Lambda 함수를 실행할 수 있는 격리된 컴퓨팅 환경을 할당하고 코드를 실행한다.
- 이 단계가 기존 애플리케이션 서버(WAS) 내부에 구현된 비즈니스 로직 계층을 대체한다. 
- 요청이 1,000개 동시에 들어오면 Lambda 함수 역시 독립적으로 1,000개가 병렬로 생성되어 처리되며, 연산이 종료되는 즉시 해당 컴퓨팅 리소스는 소멸한다.

### 4. 데이터 저장 및 조회 (DynamoDB)
- **데이터 영속화 작업:** 로직을 수행 중인 Lambda 코드가 데이터를 저장, 수정, 조회해야 할 경우, 내부 네트워크를 통해 DynamoDB에 쿼리를 전송한다.
- DynamoDB는 고성능 NoSQL 데이터베이스로서 트래픽 규모에 맞춰 자동으로 프로비저닝된 용량을 조절하며, 10밀리초 미만의 지연 시간으로 데이터를 처리한 뒤 결과를 Lambda로 반환한다.

### 5. 응답 반환 (전체 흐름의 종료)
- 데이터베이스 작업이 완료되고 Lambda 함수의 최종 실행 결과가 도출되면, 그 결과(JSON 데이터 등)를 다시 API Gateway로 반환한다.
- API Gateway는 전달받은 데이터를 정형화된 HTTP 응답(Response)으로 변환하여 최종적으로 사용자(클라이언트)에게 전달한다.
