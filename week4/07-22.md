전반 1시간은 수업, 후반 2시간은 조별 질의응답이 진행되었는데, 기술적인 부분 이외에도 많은 인사이트를 얻을 수 있었다. 가령,

1. 탁상공론하는 시간을 줄이고 직접 해보는 시간을 가져야 한다.
2. 한 명이 작업하는 것을 나머지 인원이 쳐다보고만 있는 상황은 비효율적이다. 동시에 처리할 수 있는 업무는 없을지 고민해 보아야 한다.
3. AI Driven 개발이 되어가는 요즘, AI에게 작업을 맡기더라도 안심할 수 있는 구조를 갖추는 것이 중요해지고 있다.
4. 실무에서는 단순 프롬프트나 스킬을 넘어, 이들의 조합인 워크플로우를 활발히 사용하고 있다.

모든 내용이 유익했지만 특히 3번이 가장 와닿았는데, 그 이유는 내가 AI를 활용하면서 늘 고민하던 부분이었기 때문이다.

초기에 여러 시도를 거친 결과, 그동안 안심할 수 있는 구조 하에 AI를 활용하긴 했지만 코드 레벨에서 그렇지는 못했던 것 같다. 단지 내가 수동으로 제어하면서, 해당 작업이 미치는 영향의 범위 등을 숙지하고 있었을 뿐이다.

수동으로 제어하는 것은 정신적으로도 많이 피로하고, 프로젝트 규모가 커지면 누락되는 부분이 생길지도 모른다. 특히 요즘 더 고민하던 사안이었는데, 관련 내용을 언급해주셔서 좋은 아이디어를 얻을 수 있었던 것 같다.

오늘 수강했던 원데이 클래스는 표면적으로는 프론트엔드 강의였으나, 백엔드 업무에 있어서도 고민해 보면 좋을 부분들이 많았다고 생각한다.

백엔드 개발자라고 해서 백엔드 업무만 알아서는 안 된다. 깊이는 다를 수 있겠지만, FE, DB, 인프라 등 관련 내용에 대한 학습도 꾸준히 진행하자.

# 3. 학습 내용
## GIT
### Branch와 Worktree
- **Branch (브랜치):** 
  - 하나의 작업 디렉토리(Working Directory) 내에서 커밋의 논리적인 흐름을 분리한다.
  - 브랜치를 변경(`checkout` 또는 `switch`)하면, 현재 디스크의 폴더 내 파일 상태가 해당 브랜치의 상태로 즉시 덮어씌워진다.
- **Worktree (워크트리):** 
  - 하나의 로컬 Git 저장소(Repository)에 여러 개의 독립적인 물리적 작업 디렉토리를 연결한다. 
  - 각 폴더(워크트리)마다 서로 다른 브랜치를 동시에 체크아웃 상태로 열어둘 수 있다.

### Worktree는 언제 사용할까?
- 현재 브랜치에서 코드를 작성하던 중 커밋하기에는 미완성된 상태인데, 긴급한 버그(Hotfix)를 수정하거나 다른 브랜치의 코드를 로컬 환경에서 실행해 보아야 할 때 유용하게 사용된다.
- 기존에는 이러한 상황에서 `git stash`로 작업 내역을 임시 저장하거나 저장소를 통째로 다른 폴더에 다시 `git clone`해야 했다. 
- 하지만 worktree를 사용하면 기존 폴더의 작업 상태를 그대로 둔 채, 새로운 물리적 폴더를 생성하여 즉시 다른 브랜치 작업을 시작할 수 있다.

### Worktree 사용 흐름 예시
현재 `project`라는 폴더에서 `feature-a` 브랜치를 작업 중이며, 긴급하게 `main` 브랜치를 기반으로 버그 수정을 해야 하는 상황이라 하자.
1. **새 워크트리 생성:** 
   - 상위 디렉토리에 새로운 물리적 폴더 생성.
   - 현재 작업 중인 리포지토리에서 아래 명령어를 입력하여, 기존 폴더와 별개의 새로운 폴더(`hotfix-project`)를 생성하고 `main` 브랜치를 체크아웃한다.
   ``` bash
   git worktree add ../hotfix-project main
   ```
2. **새로운 워크트리 폴더로 이동:**
   - 새롭게 생성된 디렉토리로 이동한다. 
   - 이 디렉토리는 별도의 프로젝트 폴더처럼 보이지만, 원본 리포지토리의 Git 히스토리를 모두 공유한다.
   ``` bash
   cd ../hotfix-project
   ```
3. **작업 및 커밋:**
   - 긴급한 버그를 수정하고 평소와 동일하게 커밋과 푸시를 진행한다.
   ``` bash
   git add .
   git commit -m "fix: 긴급 버그 수정"
   git push origin main
   ```
4. **원래 작업 디렉토리로 복귀:**
   - 버그 수정이 완료되었으므로, 원래 작업하던 폴더로 다시 돌아간다. 
   - 이동 전의 미완성된 작업 내역이 그대로 남아있다.
   ``` bash
   cd ../project
   ```
5. **사용 완료된 워크트리 삭제:**
   - 더 이상 필요 없는 워크트리(물리적 폴더)를 명령어 입력으로 안전하게 정리한다.
   ``` bash
   git worktree remove ../hotfix-project
   ```

## 인프라 및 배포 관련
프론트엔드 결과물을 사용자에게 전달하기 위해 사용하는 AWS 서비스들이다.

### S3 (Amazon Simple Storage Service)
- S3는 인터넷 기반의 객체 스토리지 서비스다. 
- 웹사이트를 구성하는 정적 파일(HTML, JS, CSS, 이미지 등)을 원본 그대로 저장하고 호스팅하는 역할을 한다. 
- 프론트엔드 프로젝트를 빌드하여 생성된 결과물 디렉터리(`dist/`)를 이곳에 업로드하여 보관하게 된다.

### CloudFront
- CloudFront는 콘텐츠 전송 네트워크(CDN) 서비스다. 
- S3에 저장된 정적 파일을 전 세계에 위치한 여러 엣지 서버(Edge Server)에 복사하여 캐싱한다.
- 접속 빈도가 높은 콘텐츠를 캐싱하여 가속화하는 것이 주된 목적이며, 사용자가 웹사이트에 접속할 때 원본 S3 서버까지 가지 않고 사용자와 가장 가까운 엣지 서버에서 데이터를 응답받게 하여 로딩 속도를 크게 향상시킨다. 
- 또한, HTTPS 보안 연결과 커스텀 도메인 연동을 지원한다.
- 캐싱, 보안 등 여러 이유로 인해, S3에 직접 접근하지 않고 CloudFront를 통해 정적 자원을 획득한다.

## 프론트엔드 핵심 및 빌드 도구
실제 화면을 그리고 프로젝트를 실행 가능한 형태로 묶어주는 기본 도구들이다.

### React
- 사용자 인터페이스(UI)를 구축하기 위한 자바스크립트 라이브러리다. 
- 복잡한 화면을 독립적이고 재사용 가능한 컴포넌트 단위로 분리하여 개발 및 유지보수의 효율성을 높인다.

### TypeScript
- 자바스크립트(JS)에 변수와 함수의 데이터 타입을 명시하는 기능이 추가된 프로그래밍 언어다. 
- 컴파일(코드를 실행하기 전) 단계에서 타입 오류나 누락을 검증할 수 있다.

### Vite
- 프론트엔드 개발 환경을 구성하고, 최종 코드를 가볍고 빠르게 빌드해 주는 도구다. 
- 프로젝트 시작 속도와 소스 코드 변경 시 화면 갱신 속도가 빠른 편에 속한다.

## 라우팅 및 상태 관리 도구
데이터의 흐름과 화면의 이동을 제어하는 라이브러리들이다.

### TanStack Router
- TypeScript 기반 환경에 최적화된 라우팅(페이지 이동) 라이브러리다. 
- 파일 시스템을 기반으로 경로를 자동 인식하며, 특정 페이지로 이동할 때 전달되는 파라미터들의 타입 안정성을 엄격하게 보장한다.

### TanStack Query
- 프론트엔드에서 서버 데이터를 조회, 캐싱, 동기화, 업데이트하는 일련의 과정을 관리하는 라이브러리다. 
- 데이터를 언제 다시 불러올지, 로딩 중이거나 에러가 났을 때 화면을 어떻게 처리할지를 표준화된 방식으로 제공한다.

## API 자동화 및 스타일링 도구
백엔드와의 협업을 자동화하고, 디자인을 구현하는 도구들이다.

### Orval
- 백엔드 서버의 OpenAPI(Swagger) 명세서를 읽어들여, 프론트엔드에서 사용할 TypeScript 타입 정의와 서버 통신용 코드(TanStack Query 훅)를 자동으로 생성해 주는 도구다. 
- 백엔드에서 명세서를 꼼꼼하게 작성할수록, 프론트엔드에서는 오타나 타입 불일치 없이 정확하고 안전하게 API를 호출할 수 있다.

### Tailwind CSS
- 미리 정의된 CSS 클래스(예: `text-center`, `p-4`, `flex`)를 HTML 요소나 React 컴포넌트에 직접 조합하여 스타일을 적용하는 유틸리티 우선(Utility-first) 프레임워크다. 
- 별도의 CSS 파일을 작성할 필요 없이 직관적이고 빠르게 디자인을 구성할 수 있다.

### Shadcn UI
- Tailwind CSS를 기반으로 디자인된 UI 컴포넌트 모음이다. 
- 일반적인 라이브러리처럼 패키지로 설치하는 것이 아니라, 버튼, 모달, 입력창 등의 컴포넌트 코드를 내 프로젝트로 직접 복사하여 요구사항에 맞게 내부 코드를 수정하여 사용하는 방식이다.

## 전체 동작 흐름
### 1. API 동기화
- 백엔드 개발자가 API 코드를 작성하고 OpenAPI 명세(`@NotNull`, `@Schema` 등)를 상세히 구성한다. 
- 프론트엔드 환경에서 `pnpm orval` 명령어를 실행하면, 백엔드 명세와 정확히 일치하는 TypeScript 타입과 TanStack Query 통신 코드가 자동 생성된다.

### 2. UI 개발 및 라우팅
- React 컴포넌트를 만들고, Tailwind CSS와 Shadcn UI를 활용해 화면을 디자인한다. 
- 페이지 간의 이동 규칙은 TanStack Router를 통해 타입에 안전하게 설정한다.

### 3. 데이터 바인딩
- 사용자가 특정 페이지에 진입하면, 앞서 Orval이 생성해 둔 TanStack Query 훅을 호출하여 서버로부터 필요한 데이터를 가져오고 화면에 표시한다.

### 4. 빌드 및 배포
- 개발이 완료되면 로컬이나 CI 환경에서 Vite를 이용해 `pnpm build`를 실행한다.
- 생성된 정적 파일 묶음(`dist/`)을 S3 객체 스토리지에 업로드한다.
- CloudFront(CDN)의 기존 캐시 데이터를 무효화(Invalidation)하여 갱신한다.
- 백엔드를 재빌드하거나 중단할 필요 없이, 즉시 사용자에게 새로운 프론트엔드 화면이 서비스로 반영된다.
