## 작업 과정에서 어려웠던 점 & 배운 점
### 낯선 환경에서의 프로젝트 진행
이번 메인 프로젝트는 내게 있어 정말 모든 것이 새롭다.

기존 프로젝트에서는 항상 고정 멤버가 포함되었기 때문에 프로젝트 진행 방식도 익숙했고, 업무도 거의 백엔드 위주로 배정받았으며, 팀장으로서 그렇게 특별한 역할을 수행하지 않았다.

하지만 이번 메인 프로젝트는 새로움의 연속이다. 가령,
1. 3주간 진행했던 직무 교육/스쿼드 세션에서조차 만나지 않았던, 완전 초면인 사람들로 구성
2. 단순히 백엔드만 구현하고 끝이 아니라, 기획부터 디자인/프론트엔드/백엔드/인프라까지 프로젝트 전반을 담당
3. 자발적이긴 했지만 낯선 환경에서의 팀장 역할 배정
4. 익숙한 프로젝트 협업 도구인 노션이나 슬랙이 아닌, GitHub Wiki/Projects/Discussions/Issues 채택

그렇기 때문일까, 첫 날부터 많은 난관이 있었다. 가령,
1. 본격적인 구현에 앞서 어떤 내용들을 미리 정해둬야 할지
2. 어떻게 해야 회의가 방향성을 잃지 않고 효율적으로 진행되는 것인지
3. 새로운 협업 도구들은 어떻게 사용하는 것인지
4. 팀원별로 상이한 의사소통/업무 진행 방식을 어떻게 조율해야 할지

매 순간이 당황스럽고 난처했지만, 마음가짐을 바꾸니 한결 편한 자세로 프로젝트에 임할 수 있었다.

'처음이라면 누구나 겪게 되는 현상이다. 부족한 부분은 그때그때 채워가면 되니까, 너무 완벽하게 시작하려 하지 말자.'

프로젝트를 진행하다 보면, 분명 조율이 잘 안 되거나, 의사가 제대로 전달되지 않거나, 서로가 이해한 방식이 달라서 오해가 발생하는 등 여러 문제가 발생할 수 있고, 그럴 것이라고 생각한다.

그때마다 상황을 어영부영 넘기는 대신, 문제를 돌아보고 적절한 해결책은 무엇일지 고민하는 과정을 착실히 거친다면, 오히려 순탄한 프로젝트보다 많은 것을 배우게 될 것이다. 

발생할 혹은 발생하지 않은 문제를 두려워하지 말고 지금 할 수 있는, 해야 된다고 생각하는 것들부터 하나씩 해보자.

부족한 것은 그때그때 채워가면 된다.

# 2. 종합 회고
## 잘한 점
### 한정된 시간을 효율적으로 사용하고자 노력했던 것
- 프로젝트 기간은 5주, 아니 사실상 4주로 한정되어 있기 때문에 하루하루 시간을 효율적으로 사용하는 것이 중요하다.
- 프로젝트에 대한 이해를 높이기 위해서라면 사실 모든 과정을 직접 하는 것이 좋지만, 앞서 언급한 시간적 제약으로 인해 AI에게 위탁할 수 있는 부분은 위탁하고, 중간중간 작업 내용을 검증하는 방식으로 진행했다.
- 덕분에 몇 시간, 아니 어쩌면 며칠은 더 소요되었을 수 있는 기획을 빠르게 일단락할 수 있었다.

## 아쉬웠던 점
### 일부 기록이 누락되었다
- 회의에서 다뤘음에도 기록해두지 않아, 이후 복기하는 과정에서 깊이가 얕아진 항목들이 있다.
- 심지어는 누락된 기록도 있다.

## 개선할 점
### 즉각 기록하자
- 회의 내용을 간단하게나마 바로바로 기록하자.
- 핵심 키워드만이라도 적어둔다면, 깊이도 보존하고 누락되는 정보 또한 최소화할 수 있다. 

# 3. 학습 내용
## 클라이언트 환경
### 1. 웹 앱
- 스마트폰이나 PC의 웹 브라우저를 통해 실행되는 애플리케이션이다.
- **특징:** 디바이스에 별도로 설치할 필요가 없으며, URL을 통해 접근한다.
- **장점:** 서버에서 코드를 배포하는 즉시 클라이언트에 업데이트가 반영되며, 특정 OS 환경에 구애받지 않는다.
- **단점:** 디바이스의 하드웨어 기능(카메라, 블루투스 등)이나 OS API 접근에 제한이 있으며, 오프라인 상태에서는 동작이 제한적이다.

### 2. 네이티브 앱
- 특정 운영체제(iOS, Android) 환경에 맞춰 전용 언어(Swift, Kotlin 등)로 개발된 앱이다.
- **특징:** 앱 스토어를 통해 다운로드 및 설치 과정을 거쳐야 한다.
- **장점:** OS가 제공하는 API와 하드웨어 리소스를 100% 직접 제어할 수 있어 성능이 가장 뛰어나고 렌더링 반응 속도가 빠르다.
- **단점:** iOS와 Android 버전을 각각 독립적으로 개발해야 하므로 리소스가 많이 소모되며, 코드 수정 시 매번 스토어 심사 및 업데이트 배포 과정을 거쳐야 한다.

### 3. 웹뷰 컴포넌트
- 네이티브 앱 내부에 웹 페이지를 렌더링할 수 있도록 제공되는 UI 컴포넌트다. 앱 내부에 내장된 브라우저 엔진이라고 볼 수 있다.
- **특징:** iOS의 `WKWebView`, Android의 `WebView` 클래스가 대표적이다.
- **용도:** 앱 내에서 외부 링크를 띄우거나, 약관 페이지, 자주 변경되는 이벤트 화면 등 네이티브로 구현하기 비효율적인 화면을 서버의 웹 페이지로 대체하여 렌더링할 때 주로 사용한다.

### 4. 하이브리드 앱
- 앱 컨테이너 내부에 웹 브라우저(웹뷰)를 전체 화면으로 띄우고, 주요 화면을 웹 기술로 구현한 앱이다.
- **특징:** 화면 구성에 웹 개발 코드(HTML, CSS, JS)가 그대로 들어가며, 기기는 이를 고유의 UI가 아닌 '하나의 큰 인터넷 창'으로 인식한다.
- **장점:** 웹 코드베이스 하나로 iOS와 Android 앱을 동시에 만들 수 있어 개발 속도가 빠르고 유지보수가 쉽다.
- **단점:** 사용자가 조작할 때 웹페이지를 클릭하는 것과 같아, 네이티브 앱 특유의 부드러운 화면 전환이나 즉각적인 터치 반응을 기대하기 어렵다.

### 5. 크로스 플랫폼 앱
- 하이브리드 앱과 달리 웹뷰를 전혀 사용하지 않고, 하나의 코드로 각 운영체제의 실제 네이티브 UI를 렌더링하는 앱이다. 
  - 예) React Native, Flutter
- **특징:** 프레임워크가 작성된 코드를 iOS의 'Apple 공식 요소', Android의 'Google 공식 요소'로 변환하여 기기가 진짜 자기 자신의 화면 요소로 인식하게 만든다.
- **장점:** 렌더링된 요소들이 기기의 실제 UI이므로, 웹뷰를 거치지 않아 네이티브 앱과 거의 동일하게 빠르고 부드럽게 작동한다.
- **단점:** 새로운 OS 기능이 나오면 프레임워크가 업데이트될 때까지 기다려야 하며, 경우에 따라서는 네이티브(Swift, Kotlin) 개발 지식이 추가로 필요할 수 있다.
