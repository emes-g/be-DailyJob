## 작업 과정에서 어려웠던 점 & 배운 점
### 메인 프로젝트 기획 구체화하기
기획이 정해져 있어 구현에 전념할 수 있던 웹 서버 미션과는 달리, 메인 프로젝트는 기획부터 우리 몫이다.

기획이 엉성하면 개인 프로젝트에서조차 많이 헤매게 되는데, 팀 프로젝트인 지금은 그 파급이 더 클 것으로 예상되어, 오늘은 기획 구체화에 전념했다. 

기능 명세서와 피그마를 하나하나 대조하며 다양한 항목에 대한 논의를 진행했다. 가령,
- 누락된 기능/화면은 없는지
- 오기재된 기능이나 불필요한 컴포넌트는 없는지
- 특정 기능을 우리 프로젝트에 적용할 것인지, 적용한다면 어떻게(그대로/변형해서) 적용할 것이며 그 이유는 무엇인지

예상보다 고려되지 않은 항목이 많아서, 만약 오늘 보완하지 않았다면 구현 과정에서 상당한 난항을 겪었을 수 있겠다고 생각했다. 

하지만 일각에서는 모든 인원이 투입될 필요가 있었는지, 진도가 느린 것은 아닌지에 대한 의견이 제기되었다.

나는 프로젝트 진행에 있어 필수불가결한 작업이라고 생각했지만, 내가 그렇다고 해서 모두가 그렇게 생각하지는 않는다는 것이었다.

그래서 퇴근길에 이에 대해 계속 고민하였고, 결국 '필요하고 모두가 참여해야 하긴 했지만, 그렇다고 오랜 시간을 소요해야만 하는 작업도 아니었다'라는 결론에 다다를 수 있었다. 

내가 맞다고 생각하는 것도, 누군가에겐 아닐 수 있기 때문에 보다 열린 자세로 프로젝트에 임해야 겠다는 교훈을 얻은 경험이었다.

# 2. 종합 회고
## 잘한 점
### 꼼꼼한 기록
- 지난 회고에서는 기록 누락에 대한 아쉬움을 표했으나, 오늘은 다룬 안건들에 대해 꼼꼼히 기록하여 누락이 발생하지 않았다.
- 다룬 내용을 간단하게나마 바로바로 적어두는 습관은 앞으로도 유지하고 싶다.

## 아쉬웠던 점
### 회의에 쏟았어야 할 집중이 분산되었다
- 기록에 집중하는 나머지 회의 내용을 놓쳐서, 한 번 더 말해줄 수 있는지 요청하였다.
- 문제가 반복되면 회의 지연으로 연결될 수 있기 때문에, 좋지 않다고 생각했다.

## 개선할 점
### 말할 때는 경청하자
- 기록을 하더라도, 다른 사람이 말할 때는 경청하려고 노력하자.
- 쏟아지는 정보가 많을 경우에는 키워드만 입력해두고, 발화가 끝난 이후에 간단히 정리해두면 될 것 같다.  

# 3. 학습 내용
## AWS
### IAM (Identity and Access Management)
- **Account (계정):**
  - AWS 리소스가 생성되고 관리되는 최상위 논리적 컨테이너이자 비용 청구의 기본 단위이다.
  - 각 계정은 고유한 12자리 ID로 식별되며, 서로 다른 계정의 환경은 물리적, 논리적으로 완전히 격리되어 있어 명시적인 권한 부여 없이는 상호 접근이 불가능하다.
- **User (사용자):**
  - AWS 환경 내에서 특정 인물이나 애플리케이션을 식별하기 위해 생성하는 엔티티다. 
  - AWS Management Console에 접속하기 위한 비밀번호나, API 호출을 위한 액세스 키(Access Key) 같은 장기적인 자격 증명을 가지며, 부여된 권한 범위 내에서만 작업을 수행할 수 있다.
- **Group (그룹):**
  - 다수의 User를 하나의 단위로 묶어 권한을 일괄적으로 관리하기 위한 집합이다. 
  - 그룹 자체는 자격 증명을 가지지 않지만, 그룹에 권한을 부여하면 해당 그룹에 속한 모든 User가 그 권한을 동일하게 상속받는다. 
  - 조직 내 특정 부서나 직무별로 권한을 통제할 때 사용한다.
- **Role (역할):**
  - 특정 사용자나 그룹에 영구적으로 종속되지 않고, 임시 자격 증명(Temporary Security Credentials)을 통해 리소스에 대한 권한을 일시적으로 위임하기 위해 사용하는 엔티티다. 
  - 사용자뿐만 아니라 EC2, Lambda와 같은 AWS 서비스 혹은 외부 연동 사용자가 필요한 순간에만 권한을 획득(Assume Role)하여 작업을 수행하도록 할 때 사용한다.
- **Policy (정책):**
  - AWS 리소스에 대한 접근 권한(허용/거부)을 정의한 JSON 형식의 문서다. 
  - 이 정책 문서를 User, Group, Role에 연결(Attach)함으로써 해당 주체가 어떤 서비스의 리소스에 접근하여 어떤 액션(예: 읽기, 쓰기, 삭제)을 수행할 수 있는지 상세하게 제어한다.
  - 인증과 관련된 User, Group, Role와는 달리, 유일하게 인가와 관련된 항목이다.

### Global Infra
- AWS가 전 세계에 클라우드 컴퓨팅 환경을 제공하기 위해 구축한 물리적 설비 단위다.
- **Region:**
  - 여러 개의 가용 영역(AZ)이 모여 있는 독립적인 지리적 영역(국가 단위 또는 대륙 주요 거점 단위)이다. 
  - 리전 간에는 데이터나 리소스가 자동으로 복제되지 않으며, 인프라의 기준점이 된다. (예: ap-northeast-2 / 서울 리전)
- **Availability Zone (AZ):**
  - 단일 리전 내에 존재하는 1개 이상의 물리적 데이터 센터 묶음이다. 
  - 각 AZ는 화재, 정전 등의 재난이 다른 AZ에 영향을 미치지 않도록 전력망, 통신망이 완전히 분리되어 수십 킬로미터 떨어져 구축된다.
  - 사용자는 여러 AZ에 리소스를 분산 배치(Multi-AZ)하여 시스템의 고가용성을 확보할 수 있다.
- **Edge Location:**
  - 리전이나 AZ와는 독립적인 목적을 가지며, 최종 사용자에게 데이터를 더 낮은 지연 시간(Latency)으로 전송하기 위해 전 세계 주요 도시에 조밀하게 분산 배치된 인프라다. 
  - 주로 정적 콘텐츠를 캐싱하여 빠르게 전송하는 CloudFront(CDN)나, 빠른 DNS 응답을 위한 Route 53 서비스가 이 엣지 로케이션 서버들을 활용하여 동작한다.

### VPC (Virtual Private Cloud)
- AWS 클라우드 내에 논리적으로 완전히 격리된 사용자만의 가상 네트워크 공간이다. 
- 사용자가 직접 IP 주소 범위(CIDR), 서브넷, 라우팅 테이블 등을 정의하여 네트워크 환경을 구축하고 제어하는 가장 기본적인 뼈대 역할을 한다.

### Subnet
- VPC에 할당된 전체 IP 주소 범위를 더 작은 단위로 분할한 네트워크 영역이다.
- 트래픽의 목적에 따라 리소스를 분리하기 위해 사용하며, 크게 두 가지로 나뉜다.
- **퍼블릭 서브넷 (Public Subnet):** 인터넷과 직접 통신이 가능한 서브넷이다.
- **프라이빗 서브넷 (Private Subnet):** 인터넷에서 직접 접근할 수 없도록 차단된 보호된 서브넷이다.

### EC2 (Elastic Compute Cloud)
- AWS에서 제공하는 가상 컴퓨팅 환경, 즉 가상 서버(VM)다. 
- 서브넷 내부에 배치되어 실제 애플리케이션 코드를 실행하거나 데이터베이스를 운영하는 등 실질적인 컴퓨팅 연산 작업을 수행한다.

### IGW (Internet Gateway)
- VPC 내의 리소스들이 외부 인터넷과 통신할 수 있도록 연결해 주는 관문이다. 
- 퍼블릭 서브넷이 인터넷과 양방향으로 데이터를 주고받기 위해서는 반드시 VPC에 IGW가 연결되어 있어야 한다.

### NAT Gateway (Network Address Translation)
- 프라이빗 서브넷에 위치한 리소스(예: 백엔드 EC2)가 외부에 있는 인터넷 서비스(예: 패키지 업데이트, 외부 API 호출)에 접근할 수 있도록 내부 IP를 퍼블릭 IP로 변환해 주는 서비스이다. 
- 외부에서 프라이빗 서브넷으로 직접 들어오는 인바운드 연결은 차단하면서, 아웃바운드 트래픽만 허용할 때 사용한다.

### Route Table (라우팅 테이블)
- 네트워크 트래픽이 향해야 할 목적지(경로)를 결정하는 규칙들의 집합이다. 
- 각 서브넷은 라우팅 테이블과 연결되며, 서브넷 내부에서 발생한 트래픽이 VPC 내부로 갈지, 인터넷 게이트웨이(IGW)로 갈지, NAT 게이트웨이로 갈지 지정하는 이정표 역할을 한다.

### Security Group (보안 그룹)
- EC2 인스턴스 수준에서 트래픽의 유입(인바운드)과 유출(아웃바운드)을 제어하는 가상 방화벽이다. 
- 트래픽을 허용(Allow)하는 규칙만 설정할 수 있으며, 상태를 저장(Stateful)하기 때문에 인바운드 규칙에서 허용된 트래픽에 대한 응답은 아웃바운드 규칙과 무관하게 자동으로 허용된다.

### NACL (Network Access Control List)
- VPC의 서브넷 수준에서 유입(Inbound) 및 유출(Outbound)되는 트래픽을 제어하는 방화벽이다.
- EC2 인스턴스 단위로 적용되는 보안 그룹과 달리, 서브넷 단위로 적용된다.

### VPC 아키텍처 플로우
1. **네트워크 기반 생성 (VPC & 서브넷):**
   - 먼저 VPC를 생성하여 전체 네트워크 영역을 확보한다. 
   - 그 후 보안 목적에 따라 네트워크를 나눈다. 
   - 외부에서 접근해야 하는 리소스를 위한 퍼블릭 서브넷과, 외부 접근을 차단해야 하는 Spring Boot 애플리케이션 및 MySQL DB를 위한 프라이빗 서브넷을 각각 생성한다.
2. **인터넷 연결 구성 (IGW & NAT Gateway):** 
   - VPC에 IGW를 부착한다. 
   - 그리고 프라이빗 서브넷의 리소스가 외부 통신을 할 수 있도록 퍼블릭 서브넷 안에 NAT Gateway(혹은 인스턴스)를 배치한다.
3. **트래픽 경로 지정 (Route Table):**
   - 퍼블릭 서브넷의 라우팅 테이블에는 "외부 인터넷으로 가는 트래픽은 모두 IGW로 보내라"는 규칙을 추가한다.
   - 프라이빗 서브넷의 라우팅 테이블에는 "외부 인터넷으로 가는 트래픽은 모두 NAT Gateway로 보내라"는 규칙을 추가한다.
4. **컴퓨팅 리소스 배치 (EC2):** 
   - 각 서브넷에 가상 서버인 EC2를 배치한다.
5. **접근 제어 (Security Group):**
   - 퍼블릭 서브넷의 EC2(예: 웹 서버)에 적용된 보안 그룹은 외부 사용자의 HTTP(80) 및 HTTPS(443) 포트 접근을 허용한다. 
   - 프라이빗 서브넷의 EC2(예: 백엔드 서버)에 적용된 보안 그룹은 오직 퍼블릭 서브넷에 있는 특정 보안 그룹으로부터 유입되는 트래픽(예: 8080 포트)만 허용하도록 설정하여 보안을 극대화한다.
