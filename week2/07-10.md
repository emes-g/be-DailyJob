## 작업 과정에서 어려웠던 점 & 배운 점
### 직무별 밋업
현대/기아 ICT 본부 소속 책임님 1분, 매니저님 2분께서 오셔서 부서 소개 및 질의응답을 해주셨다.

2시간 가량 진행됐는데 대부분의 시간이 질의응답으로 진행되었고, 부서 소개를 통해서는 '어떤 부서에서, 어떤 고객층을 대상으로, 어떤 서비스를 제공하고 있는가' 정도를 대략적으로 이해할 수 있었다.

질의응답은 사전질문 및 현장 추가질문으로 구성되었는데, 많은 인사이트를 얻을 수 있는 시간이었다.

신입 개발자가 가져야 할 마인드셋은 어떠한지, 이슈가 발생했을 때 어떤 프로세스로 해결해야 할지, AI를 어떻게 활용하고 있는지, 의견 충돌이 발생했을 때 합리적으로 해결할 수 있는 방안은 무엇일지, 일을 잘한다고 평가받는 사람들의 공통된 특징은 무엇인지 등 다양한 주제에 대해 성심성의껏 답변해 주셨다.

답변을 들으면서 느꼈던 점은, 실무라고 해서 특별히 다른 점은 없다는 것이었다.

1. 실수는 누구나 할 수 있지만, 그 이후의 대처 과정이 중요하다. 실수했다는 사실을 숨겨서는 안 되며, 동일한 실수를 반복하지 않도록 문제 발생부터 해결까지 일련의 과정을 돌아보아야 한다.

2. 이슈 발생 원인을 빠르게 파악하고, 관련 담당자와 소통하여 해결해야 한다.

3. 수작업으로 진행하던 테스크 중 비효율적이었던 부분들을 AI로 개선하고 있다.

4. 의견 충돌 자체는 유의미한 경험이며, 의견 충돌이 발생했을 때는 논리에 입각하여 해결해야 한다.

5. 단순히 주어진 일만 수행하는 것이 아니라, 주도적으로 수행할 수 있는 업무를 찾아 나서는 사람들이 보통 일을 잘한다고 평가받는다.

업무를 효율적으로 수행하기 위해 개발자, 나아가 현업에서 지켜야 할 수칙들이었다.

당연하면서도, 그 순간이 되어 막상 적용하려고 보면 쉽지 않은 항목들이기 때문에, 평소에 내재화시켜둬야 겠다는 생각을 하게 된 시간이었다. 

### `ERR_EMPTY_RESPONSE` 응답이 반환되었을 때, 서버에 관련 로그나 StackTrace가 출력되지 않았던 문제
'로그인 유저가 로그인/회원가입 페이지로 이동하지 못하도록 제한하는 기능'을 개발하여 테스트하던 중, 브라우저에서 `ERR_EMPTY_RESPONSE` 응답이 반환되는 문제를 겪었다.

테스트 과정에서 문제가 발생할 수 있는 것은 당연했지만, 진짜 문제는 서버에 관련 로그나 StackTrace를 찾을 수 없다는 것이었다.

그래서 현재 요청 흐름을 제어하고 있는 `RequstHandler.processRequest()`에서 어디까지 실행되는가를 먼저 확인했고, 매핑 컨트롤러 조회에서 `null`이 반환되어 요청 처리 위임(`controller.service()`)이 이루어지지 못하고 있는 상황임을 인지할 수 있었다.

요청 메시지 파싱 여부, 요청 정상 처리 여부, 특정 상황(`MethodNotAllowed`, `UnAuthorized`)에 대한 예외는 로깅하고 있었지만, `NPE`는 해당 스레드가 죽는 것이 맞다고 판단했기 때문에 의도했던 상황이었다.

다만 스레드가 죽었다면, 필연적으로 나타났어야 할 StackTrace는 왜 보이지 않았을까?

문제를 해결하기 위해, 요청 흐름을 제어하기 전 프로그램이 어떤 방식으로 동작하고 있는가를 확인했더니 답을 찾을 수 있었다.

현재 프로그램은 `WebServer`의 메인 스레드로 요청이 들어오면 스레드 풀에 작업(`RequestHandler.run()`)을 위임하는 구조로 동작하고 있었는데, 위임할 때 `execute()`가 아닌 `submit()`을 활용하고 있었기 때문이다.

에러가 발생하면 스레드를 폐기하는 `execute()`와 달리, `submit()`은 예외를 캡슐화하여 `Future` 객체에 저장하므로 StackTrace가 출력되지 않았던 것이다.

유지보수에 용이한 구조로 개발한 덕에 문제 자체를 금방 찾을 수 있었던 점은 좋았지만, 메서드 하나를 사용할 때도 주의해야 한다는 경각심 또한 가지게 된 경험이었다.

# 3. 학습 내용
## Concurrent 패키지
### ExecutorService 인터페이스
- **`execute(Runnable)`:** 반환값 없이 단순히 실행하며, 작업 도중 예외가 발생하면 해당 스레드 폐기
- **`submit(Runnable or Callable)`:**
  - 작업 상태를 추적할 수 있는 `Future` 객체를 반환
  - 에러가 발생해도 예외를 캡슐화하여 `Future` 객체에 저장
  - 스레드가 폐기되지 않고 재사용되므로 훨씬 안정적

## 동적 HTML
- 확장자가 포함된 동적 HTML 경로는 제공되지 않도록 막아야 한다.
  - 가령, 로그인 페이지를 `/login/index.html` 경로로 제공하는 것
- 정적 파일처럼 제공될 경우, 데이터가 주입되지 못해 템플릿 문법이 그대로 노출되기 때문이다.

# 4. 주간 프롬프트 회고
### 1. 내가 최종 선택한 가장 잘 작성했다고 생각하는 프롬프트
원본 프롬프트:
````text
현재 WAS 서버를 직접 구축해 보고 있는데, '인증되지 않은 사용자가 인증이 필요한 서비스를 요청하는 경우 정상적으로 응답되는 문제'가 있어.
해당 문제는 다음의 2가지 방식으로 해결해 볼 수 있을 것 같고, 내가 생각하는 각 방안의 장단점은 다음과 같아.
1. 필드로 `ThreadLocal<User>`을 가지고 있는  `AuthenticationContext` 클래스를 신설하여, 인증 정보를 저장하는 방식.
  - HttpRequest 객체의 순수성을 유지할 수 있다.
  - ‘인증된 사용자 요청 여부’ 확인 기능을 추가하는 과정에서 기존 코드를 수정하지 않아도 된다.
  - 해당 클래스 이외에도 추가적으로 생성해야 하는 클래스(`AuthenticationManager` 등)가 존재한다.
  - 스레드 풀 환경에서 사용할 경우, 메모리 누수 및 보안 사고가 발생할 우려가 있다.
2. `HttpRequest` 객체에 `attributes` 필드를 추가하여, 인증 정보를 저장하는 방식.
  - 기존 클래스만으로 작업을 진행할 수 있다.
  - HttpRequest 객체의 순수성을 잃게 된다.
현재 언급한 내용에서 틀린 부분은 있는지, 이외에도 각 방안이 보유한 장단점이 무엇인지 텍스트로 알려줘. 
````

선택한 이유: 단순히 문제 상황뿐만 아니라, 생각하고 있는 해결 방안과 그 근거를 같이 제시했다. 나아가, 해당 해결 방안과 근거가 틀릴 수 있다는 사실 또한 제공했다. 

### 2. 원본 프롬프트 구조 분석
- **문제 상황:** 인증되지 않은 사용자가 인증이 필요한 서비스를 요청하는 경우 정상적으로 응답되는 문제
- **원하는 결과를 어떻게 요구했는가:** 가져온 해결 방안에서 오개념은 없는지, 생각하지 못한 장단점은 무엇이 있는지 알려달라고 요청했다.
- **필요한 맥락이 포함되어 있는가:** WAS를 직접 구축 중이라는 환경적 맥락과 함께, `AuthenticationContext`, `ThreadLocal<User>`, `HttpRequest` 등 현재 고려 중인 구체적인 클래스가 명시되어 있다.
- **판단 기준이나 제약 조건이 있는가:** 가져온 해결 방안을 기준으로만 답변할 수 있도록, 범위를 한정했다.
- **출력 방식이 명확한가:** '텍스트로 알려줘'와 같이 출력 방식을 명확히 지정했다.
- **나의 가설, 우려, 의도가 드러나는가:** 각 방안이 가질 수 있는 문제점(`HttpRequest`의 순수성 유지 여부, 스레드 풀에서 발생할 수 있는 메모리 누수 및 보안 문제)에 대한 우려를 표했고, 오개념을 잡아줄 수 있도록 '내가 가져온 내용이 틀릴 수 있다'는 내용을 명시했다.

첫째, 제가 작성한 장단점 중 기술적으로 틀린 오개념이 있다면 팩트 체크를 해주세요.

### 4. AI가 개선한 프롬프트 분석
- **더 명확해진 목표 및 출력 형식:**
  - '틀린 부분은 있는지, 장단점은 무엇인지 알려줘'라는 기존의 요청을 1번(오개념 여부 체크)과 2번(추가 장단점 도출)으로 넘버링하여 분리했다.
  - AI가 요구사항을 누락하지 않고 구조화된 형태로 답변하도록 유도한 것이다.
- **추가된 제약 조건:**
  - '텍스트로 알려줘'라는 기존 조건에 '전체 코드를 구현할 필요는 없습니다'라는 제약 조건이 추가되었다.
  - 덕분에 AI는 불필요한 코드 작성 대신 아키텍처 분석에 집중할 수 있게 되었다. 
- **구체적 판단 기준 제시:**
  - 단순히 '추가 장단점'을 묻던 내 원본 프롬프트와 달리, '성능, 보안, 유지보수 측면으로 나누어'라는 구체적인 판단 기준이 제시되었다.
- **답변 품질이 좋아질 것으로 예상되는 이유:**
  - 출력 형식이 명확해지고 구체적인 판단 기준이 제시된 덕에, 보다 깊이 있고 체계적인 아키텍처 리뷰 결과를 기대할 수 있게 되었다.
