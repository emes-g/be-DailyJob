## 작업 과정에서 어려웠던 점 & 배운 점
### 인증 필터 로직 추가 및 라우터 리팩토링
로그인/로그아웃 기능을 구현하고 이것저것 테스트하다 보니, 한 가지 문제점을 발견할 수 있었다.

인증되지 않은 사용자가 인증이 필요한 서비스를 요청하면 서버에서는 이를 정상적인 요청으로 간주하여, 해당 서비스를 제공하는 것이었다.

이를 해결하기 위해서는 애플리케이션에 저장된 ‘인증 정보’를 바탕으로, ‘인증된 사용자 요청 여부’를 확인할 수 있는 기능을 추가해야 했다.

우선, 인증 정보를 어떻게 저장할 수 있을지 고민한 결과, 두 가지 해결 방안을 찾을 수 있었다.

첫 번째로, 필드로 `ThreadLocal<User>`을 가지고 있는  `AuthenticationContext` 클래스를 신설하여, 인증 정보를 저장하는 방식.

두 번째로, `HttpRequest` 객체에 `attributes` 필드를 추가하여, 인증 정보를 저장하는 방식.

두 방식 모두 장단점이 있었지만, 나는 `ThreadLocal` 방식을 선택했다. 기존 클래스(`HttpRequest`)의 순수성을 지키고, ‘인증된 사용자 요청 여부’ 확인 기능을 추가하는 과정에서 기존 코드를 수정하지 않아도 됐기 때문이다.

다음으로, ‘인증된 사용자 요청 여부’ 확인을 기존 요청 흐름 중 어디서 수행할지 고민했다.

매핑 컨트롤러를 찾고 난 후(`Router.getController()` 이후 시점)에도, HTTP 요청 메시지 파싱 직후에도 가능한 상황이었다.

나는 후자를 선택했는데, 그 이유는 ‘인증된 사용자 요청 여부’ 확인에 필요한 쿠키는 요청 메시지 파싱 이후 시점이라면 어디서든 조회할 수 있었고, 그렇다면 효율 상 가장 빠른 시점에 해주는 것이 맞다고 판단했기 때문이다.

두 가지 결정 사항을 적용한 결과, 인증되지 않은 사용자가 인증이 필요한 서비스를 요청할 경우 `UnAuthorizedException`을 던지고 사용자를 로그인 페이지(`/login/index.html`)로 리다이렉트시킬 수 있게 되었다.

기재한 내용 외에도, 이 문제점을 해결하는 일련의 과정에서 수많은 선택의 순간을 맞닥뜨렸다.

그때마다 각 방안의 장단점은 무엇이고, 내가 어떠한 이유에서 특정 방안을 선택했는지 근거를 확정하고 넘어가려 노력했다. 모든 결정에는 납득 가능한 이유가 있어야 한다고 생각했기 때문이다.

남은 미션/프로젝트/해커톤, 나아가 수료 후까지도 이러한 태도를 유지해야 할 것이다.

# 3. 학습 내용
## Java
### 함수형 연산자 map
- 스트림 내 데이터를 다른 형태로 추출하거나 변환할 때 사용한다.
```java
return Endpoint.findBy(path, method)    // Optional<Endpoint>
        .map(Endpoint::isAuthRequired)  // Optional<Boolean>
        .orElse(false); // boolean
```
- **`Endpoint.findBy(path, method)`:**
  - 경로와 메서드에 매칭되는 `Endpoint` 객체를 찾는다.
  - 결과가 존재할 수도, 없을 수도 있으므로 `Optional<Endpoint>`를 반환한다.
- **`.map(Endpoint::isAuthRequired)`:**
  - 값이 있다면, `Endpoint` 객체의 `isAuthRequired()` 메서드를 실행하여 결과를 `Optional<Boolean>`에 담는다.
  - 값이 없다면, 그대로 흘려보낸다.
- **`orElse(false)`:**
  - `Optional` 내부의 값을 꺼낸다.
  - 값이 없다면(`Optional.empty`) 기본값을 반환한다. 

### split(regex, limit)
- `split()` 메서드는 구분될 문자열의 개수를 지정할 수 있다.

### 접근 제한자
- **public:** 외부 클래스가 자유롭게 사용할 수 있다.
- **protected:** 같은 패키지 혹은 자식 클래스에서만 사용할 수 있다.
- **private:** 외부에서는 사용할 수 없다.
- **default:** 같은 패키지에서만 사용할 수 있다.

### ThreadLocal
- 스레드별로 데이터를 보관할 수 있는 메모리 공간이다.
- 스레드를 반환하기 전에는 해당 공간을 항상 비워줘야 한다.
  - 스레드 풀이 아니라, 매번 스레드를 생성하고 지운다면 상관은 없겠으나 거의 사용되지 않는 방식이다.
