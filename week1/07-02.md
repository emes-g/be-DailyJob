## 작업 과정에서 어려웠던 점 & 배운 점
### 어렴풋이 알아서는 적용할 수 없다
각자 상품 주문 ERD를 구축하고, 조별로 의견을 나눠보는 시간을 가지기로 했다.

앞서 모임통장 ERD를 구축한 경험도 있고, 이론은 지금껏 여러 번 들었기 때문에 ERD를 구축하는 과정이 수월할 줄 알았지만, 막상 그려지지 않았다.

지금껏 듣고 보는 것에 익숙해져 있었을 뿐, 처음부터 끝까지 스스로의 힘으로 ERD를 그렸던 경험이 없다시피 했기 때문이다.

많이 접하는 것도 분명 중요하지만, 단순히 접하는 것에 그치지 않고 직접 처음부터 끝까지 해보는 경험 또한 중요하다는 사실을 되새길 수 있었다.

개발자로서 앞으로 내가 마주하게 될 문제들은 객관식이 아닌, 서술형이기 때문이다.

수동적으로 받아들이기만 하지 말자. 깊게 고민할수록, 많은 것을 알아갈 수 있다.   

### 마주하지 않으면 해결되지 않는다
개인적으로 상품 주문 ERD를 구축하는 과정에서, '상품 엔티티에 이미 가격 속성이 존재하는데, 주문상품 엔티티에도 가격 속성이 존재해야 할까?' 라는 의문이 생겼다.

처음에는 답을 내지 못했지만, 조별 활동을 진행하는 과정에서 '상품 엔티티의 가격 속성에만 의존한다면, 가격이 변경되는 경우 데이터 정합성이 깨질 수 있겠다'는 생각이 들었다.

사실 지금껏 여러 번 겪어 봤던 문제 같은데, 매번 명확한 결론을 내지 않고 넘어갔다 보니 오늘까지도 의문으로 남아 있던 것이다. 오늘조차 미뤘다면, 나는 다음에도 이 문제를 고민하고 있었을 것이다. 

마주하지 않으면 해결되지 않는다. 당연하면서도, 와닿는 말이다.

### 익숙한 것과 아는 것은 다르다
웹 서버 1단계 미션을 진행하면서, `InputStream`, `InputStreamReader`, `BufferedReader`를 다루게 되었다.

키보드로 입력받을 때 아무렇지 않게 사용했던 `BufferedReader(new InputStreamReader(System.io))`이지만, 막상 하나씩 나눠서 생각해 보니 쉽사리 대답이 나오지 않았다.

나는 해당 클래스들이 익숙했던 거지, 알고 있다고 할 수 없던 상태였던 것이다.

지금껏 낯선 개념들은 항상 찾아 보고 이해한 후에 사용했지만, 등잔 밑이 어둡다고 오히려 익숙한 개념들은 별 생각없이 사용했던 것 같다.

익숙한 것과 아는 것은 다르다. 익숙한 키워드더라도, 주기적으로 어떤 의미를 갖는지 되새겨 보자.

# 2. 종합 회고
## 잘한 점
### 주위에 휘둘리지 않았다
- 조원이나 동기들과 비교했을 때, 웹 서버 미션 진행 속도가 비교적 느린 편이다.
- 속도가 빠르면 분명 좋겠지만, 그렇다고 깊이를 놓쳐서는 안 된다고 생각했다.
- 부족한 시간은 퇴근 후나 여가 시간을 활용할 수도 있다.
- 조급해져서 본래의 깊이를 놓치지 말자.

## 아쉬웠던 점
### 문제를 해결하는 것에 그쳤다 
- 분명 오늘 회고에 기록한 내용 이외에도 해결한 문제들이 있었는데, 정리하지 않았더니 단편적으로 기억에 남았다.

## 개선할 점
### 트러블 슈팅 내역을 기록해두기
- 어떤 문제가 왜 발생했고, 어떻게 해결했는지를 기록해 두자.
- 하루 이틀이야 동일한 문제를 마주하더라도 기억나서 금방 해결하지만, 상황과 해결 방법을 정리해두지 않으면 나중에 똑같은 트러블 슈팅을 하고 있을지도 모른다.

# 3. 학습 내용
## Java
- InputStream: 네트워크 소켓을 통해 들어오는(입력) 바이트 데이터를 읽기 위한 통로(스트림)
- InputStreamReader: 이름에서 알 수 있듯, 입력으로 들어오는 바이트 스트림을 지정된 형식으로 읽기 위한 Reader
- BufferedReader: 데이터를 효율적으로 읽을 수 있도록, 버퍼 기능을 제공해주는 Reader

## 추가적으로 학습할 내용
- 어제 회고에서도 다뤘지만, 수업을 들었다고 내 것이 되지 않는다.
- 논리 데이터 모델링 파트는 별도로 학습해서 정리해야 겠다.
