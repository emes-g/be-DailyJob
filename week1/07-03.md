## 작업 과정에서 어려웠던 점 & 배운 점
### 내가 구현하지 않은 코드를 이해하는 것
다른 사람이 작성한 코드의 전체 흐름을 파악하는 경험은 오랜만이었다.

단순히 처음부터 끝까지 코드를 읽는 것이 아닌, 구현에 담긴 의도를 추론하며 읽는 과정은 많은 공부가 되었다.

현재 구조를 채택한 이유는 무엇인지, 내 방식과 어떻게 다른지, 각각은 어떠한 장단점을 가지고 있는지, 우리의 방식을 더 발전시킬 수는 없을지 등 여러 포인트를 생각해 볼 수 있었다.

기존에는 페어 프로그래밍 방식으로 개발했던 경험이 많았다. 이는 코드의 흐름을 놓치지 않고 따라가기에는 수월하지만, 문제를 바라보는 시야 자체는 비교적 좁아지는 방식이다.

익숙한 방식은 편한 만큼 얻는 것이 적고, 새로운 방식은 과정이 고되지만 그만큼 얻는 것도 많다.

앞으로는 다양한 방식을 고루 적용하여, 전반적인 개발 역량 자체를 향상시킬 수 있도록 해야겠다.

### 이해할 수 있게 설명하는 것
조별 활동과 스쿼드 세션을 거치며, 이해할 수 있게 설명하는 것이 중요하다는 사실을 다시금 깨달았다.

상대방이 이해할 수 있게 설명하려면 어떻게 해야 할까?

정답은 없지만, 나는 흐름을 짧게 가져가는 것이 중요하다고 생각한다. 흐름이 짧아질수록 말하고자 하는 바가 명료하게 드러나기 때문에, 청자가 알고 있는 내용이라면 더 깊은 수준의 대화로 이어질 수 있고, 모르는 내용이라면 어느 부분을 풀어서 설명해야 할지 파악하기 용이해진다.

흐름을 짧게 가져가며, 지속적으로 상대방과 소통해보자. 같은 높이에서 바라보고 있어야, 나도 상대방도 많은 것을 얻어갈 수 있다.

# 2. 종합 회고
## 잘한 점
### 모르는 채 넘어가지 않았다
- 대화는 서로의 이해를 바탕으로 진행된다.
- 모르는 채 한두 번 넘어가다 보면, 어느 순간 대화에 참여하지 못하게 된다.
- 원활한 대화가 이어질 수 있도록, 지속적으로 모르는 내용을 짚고 넘어갔던 점이 좋았다고 생각한다.

## 아쉬웠던 점
### 안도감 때문에 느슨해졌다
- 기존에 레포지토리를 잘못된 방식으로 생성하여, 스쿼드 세션이 끝나고 웹 서버 미션 레포지토리를 다시 만들었다.
- 여기까지는 좋았는데, 안도감 때문인지 이후 1시간 가량은 집중하지 못했던 것이 아쉽다.

## 개선할 점
### 집중이 안 될 때는 다른 작업을 수행해 보자
- 상황에 따라 집중이 잘 되는 업무가 있고, 그렇지 않은 업무가 있다.
- 집중이 안 돼서 진척이 없을 때는, 다른 업무를 수행해 보자.

# 3. 학습 내용
## Concurrent 패키지
- 안전하고 효율적인 멀티스레드 프로그래밍을 지원하는 패키지
- 크게 Executor Framework, Concurrent Collections, Synchronizers & Atomics로 구성되어 있다.

### Executor Framework (실행자 프레임워크)
- 등장 배경: 스레드를 직접 생성하는 방식이 갖는 문제 (스레드 생성 비용, OOM, Runnable 인터페이스의 불편함 등)
- 작업의 실행 흐름과 스레드 풀 관리의 책임이 개발자에서 프레임워크로 이동되었다.

#### Executor 인터페이스
- 비동기 작업 실행의 최상위 인터페이스
- 개발자가 직접 스레드를 생성하여 실행하던 기존과 달리, '작업(`Runnable`)을 넘기면 알아서 실행한다'는 개념(작업 제출과 실행을 분리)을 처음 도입한 규격 

#### ExecutorService 인터페이스
- Executor 인터페이스를 확장하여, 작업의 생명주기 관리와 비동기 결과 추적 기능을 추가한 인터페이스
- 작업 완료 여부 확인, 스레드 풀 종료, 작업 제출 메서드 등을 제공
  - **`execute(Runnable)`**: 반환값 없이 단순히 실행하며, 작업 도중 예외가 발생하면 해당 스레드 폐기
  - **`submit(Runnable or Callable)`**:
    - 작업 상태를 추적할 수 있는 `Future` 객체를 반환
    - 에러가 발생해도 예외를 캡슐화하여 `Future` 객체에 저장
    - 스레드가 폐기되지 않고 재사용되므로 훨씬 안정적

#### Future 클래스
- 작업 결과를 기다릴 필요 없이 즉시 반환을 받고, 필요할 때만 결과를 기다릴 수 있게 유연성을 제공하는 클래스

#### Executors 인터페이스
- 다양한 종류의 스레드 풀(`ExecutorService` 구현체)을 쉽게 생성할 수 있도록 돕는 정적 팩토리 메서드
- 예) `newFixedThreadPool()`

#### ThreadPoolExecutor 클래스
- `ExecutorService`의 대표적인 구현체
- BlockingQueue와 Thread Pool을 이용하여 멀티스레드 환경에서의 작업을 효율적으로 처리

### Concurrent Collections (동시성 컬렉션)
- 멀티스레드 환경에서 데이터 정합성을 보장해주는 컬렉션
- 예) `ConcurrentHashMap`, `CopyOnWriteArrayList`, `BlockingQueue` 등

### Synchronizers & Atomics (동기화 유틸리티 및 원자적 변수)
- 특정 스레드들이 작업을 마칠 때까지 대기하거나, 락 없이도 스레드 안전하게 값을 연산할 수 있도록 지원하는 도구
- 예) `CountDownLatch`, `AtomicInteger`
