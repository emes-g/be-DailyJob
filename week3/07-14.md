## 작업 과정에서 어려웠던 점 & 배운 점
### 파일 업로드 기능 구현
7단계에서는 파일 업로드 기능을 지원해 달라는 요구사항이 있었다.

기존에는 텍스트 데이터 처리만을 고려하여 요청 메시지 파싱을 `InputStreamReader`(정확히는 `BufferedReader`)로 수행하였는데, 지원해야 하는 데이터의 종류가 늘어났으므로 요청 메시지 파싱 도구 또한 변경해야 했다.

결국 파일 업로드 기능을 구현하기 위해서는, 기존의 요청 메시지 파싱 구조부터 개선해야 했다.

파일 데이터를 지원하기 위해서는 바이트 스트림을 직접 파싱할 필요가 있었는데, `BufferedReader` 도입 당시와 유사하게 ‘버퍼를 두어서 성능 상의 이점을 챙기자’는 생각으로 `BufferedInputStream`을 선택하게 되었다.

요청 메시지 파싱 기반을 변경한 후에는 본격적으로 파일 업로드 기능을 구현하게 되었다.

단순히 지원하는 데이터를 하나 늘릴 뿐이라 금방 끝낼 수 있으리라 생각했는데, 예상 외로 ‘파일 업로드 기능’을 지원하기 위해 수반되어야 할 작업이 많았다. 가령,
1. 클라이언트가 서버로 여러 종류의 데이터를 보낼 때는 멀티파트 데이터 형식으로 전송하는데, 해당 형식을 파싱하는 일련의 과정에는 ‘Boundary 추출’, ‘하나의 멀티파트 데이터를 여러 개의 파트 데이터로 분리’, ‘파트별 Content-Disposition 추출’, ‘파트 데이터 종류에 따른 처리 방식 분기’ 등 여러 서브 작업이 요구되었다.
2. 또한 후술하겠지만, 파일 업로드 기능은 ‘마이페이지 프로필 이미지 변경’, ‘게시글 파일 첨부’ 등 다른 서비스를 지원하기 위한 기반 기능으로, 결국 위 작업을 수행하는 과정에서 프론트엔드 영역까지 수정해야 했다.

그나마 다행이었던 점은 지금까지 계속 유지보수에 용이한 구조를 고민하고, 적용해왔던 덕에 ‘파일 업로드 기능 구현 자체’에만 집중할 수 있었다는 점이다. 만약 웹 서버를 유지보수하기 어려운 구조로 설계해 왔었다면, 구조에 대한 리팩토링도 함께 진행했어야 했기 때문에 작업 난이도도 올라가고, 속도도 더욱 느려졌을 것이다.

이번에 요청 메시지 파싱 구조를 변경하고, 파일 업로드 기능을 구현한 경험을 통해, 기능 구현에 필요한 작업을 세분화하고, 유지보수가 용이한 구조로 설계하는 것이 왜 중요한지 실감할 수 있었다.

# 2. 종합 회고
## 잘한 점
### 조원의 문제를 함께 해결하기
- 아침에 출근하고 보니, 옆의 조원이 충돌 때문에 브랜치 병합에 실패하는 문제를 겪었다고 한다.
- 당장 나도 며칠 전에 upstream, fetch, rebase를 공부하기도 했고, 문제 해결 과정에서 나 또한 배울 수 있는 점이 있다고 생각해, 함께 해결해봐도 괜찮을지에 대해 여쭤본 후 도와드리게 되었다.
- 어디서 충돌이 발생했는지, 현재 로컬과 origin 브랜치는 어떤 상황인지, 어떻게 해결할 수 있을지를 같이 논의하며 학습했던 개념들을 적용해볼 수 있었다. 
- 내가 발생한 문제는 아니었지만, 나 또한 언제든지 문제가 발생할 수 있고, 문제 해결 과정에서 새롭게 얻어가는 것 또한 분명히 존재하기 때문에, 앞으로도 '문제 해결을 도울 수 있으면 좋겠다'고 생각하게 된 경험이었다. 

## 아쉬웠던 점
### 탭이 너무 많이 열려 있어서, 원하는 정보를 찾기 힘들다
- 수업을 듣거나 미션을 진행하다 보면 모르는 내용이 나오기 마련이다.
- 그때마다 탭을 열어두게 되는데, 어느새 확인해 보면 탭이 너무 많이 열려 있어서 원하는 정보를 찾기 힘들 때가 있다.

## 개선할 점
### 그때그때 정리해서 필요한 탭만 남겨두자
- 원하는 내용을 즉각 발견할 수 있으려면, 가능한 탭의 수를 적게 열어두어야 한다.
- 관련 내용 학습을 마쳤다면, 그때그때 정리해서 필요한 탭만 남겨둘 수 있도록 하자.

# 3. 학습 내용
## Java
### 패키지와 디렉터리는 다르다
- 패키지는 디렉터리 경로가 아니라 소스 코드 내부의 `package` 선언문으로 결정된다.
- `src/main/java`와 `src/test/java` 컴파일 시작점(Source Root)일 뿐이며 패키지 이름에 포함되지 않는다.
- 빌드 도구(Gradle/Maven)가 두 경로의 결과물을 같은 클래스패스에 올리기 때문에 JVM은 이를 '동일 패키지'로 인식한다.
- 애플리케이션 코드와 테스트 코드를 같은 '패키지 경로'에 위치시키는 이유 중 하나는, 테스트 과정에서 `default` 대상에 접근할 수 있도록 하기 위함이다.
