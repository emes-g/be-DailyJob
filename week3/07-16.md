# 2. 종합 회고
## 잘한 점
### AI 도구 활용 방식을 변경해 보았다
- 지난 회고에서 '기존 AI 활용 방식(대화형 AI 기반)의 한계를 실감하여, 새로운 방식(에이전트 기반)을 학습하였다'고 했는데, 오늘은 이를 적용해 보았다.
- 사실 웹 서버 미션 진행에 있어서는 무리가 없기도 하고, 버그 데이가 당일이었기 때문에 기존 방식을 고수할 수도 있었겠지만, 당장의 급한 불만 꺼서는 의미가 없다고 판단했기 때문이다.
- 새로운 기술을 학습/도입할 때마다 느끼는 점이지만, 이전의 단점을 극복하기 위해 등장한 기술이라 할지라도 모든 면에서 완벽하지는 않다는 것이다. 가령,
  - 대화형 AI 기반의 기존 방식은 컨텍스트 관리를 온전히 내가 해야 한다는 단점이 있었지만, 그 과정에서 아키텍처와 구현 코드에 대한 이해는 더 깊어질 수 있었다.
  - 반면 에이전트 기반의 새로운 방식은 컨텍스트 관리를 해 줄 필요가 없지만, 그렇기 때문에 단순히 사용하기만 해서는 '프로젝트가 현재 어디까지 와 있는지', '특정 부분은 어떤 방식으로 동작하는지', '어떤 선택지들 중에서 이 방안이 채택된 건지' 등 프로젝트 전반에 대한 이해가 부족해질 수 있다.
- 언급한 내용 외에도 여러 장단점을 가지고 있지만, 내가 느꼈던 것은 결국 모든 기술에는 빛과 그림자가 있기 때문에 맹목적으로 사용해서는 안 된다는 것이다. 특히 에이전트는 편리해지는 만큼 다방면에서 주의 깊게 사용할 필요가 있다고 생각했다.
- 앞으로도 수많은 기술이 등장하게 될텐데 단순히 '새로운 기술이니까'가 아니라, '현재 겪고 있는 문제를 해결할 수 있는지', '장점은 무엇이고 발생 가능한 사이드 이펙트는 무엇일지' 등 다각적인 관점에서 기술을 판단하는 개발자로 성장하고 싶고, 그렇게 되도록 노력할 것이다.  

## 아쉬웠던 점

- 다른 조원으로부터 웹 서버 미션 관련 질문이 들어왔는데, 알고 있는 부분이었음에도 질문 의도와는 다르게 이해해서 적절한 답변을 드리지 못했던 것 같다.

## 개선할 점
### 내가 이해한 바가 맞는지 짚고 넘어가자
- 의도치 않은 방향으로 답변을 드리면, 원하는 부분에 대한 답을 얻지 못하고 그로 인해 시간도 더욱 지체될 수 있다.
- 오해가 발생하지 않도록, 내가 이해한 부분이 맞는지 가볍게라도 확인한 후에 답변을 드릴 수 있도록 하자. 

# 3. 학습 내용
## 캐시 (Cache)
- **Write-Through (동시 기록):** 데이터를 캐시와 DB에 동시에 기록하여 데이터 일관성을 유지하는 방식이다. 
- **Write-Back (지연 기록):** 캐시에만 먼저 데이터를 쓰고, DB에는 나중(캐시 공간 부족, 특정 주기 등)에 반영하여 쓰기 성능을 극대화하는 방식이다.

## 디바운싱 (Debouncing)
- 버튼이 눌리거나 화면이 스크롤될 때 발생하는 수많은 신호 중, 마지막에 발생한 단 한 번의 신호만 유효하게 처리하는 기술이다.
- 불필요한 중복 처리를 방지하여 시스템의 오류를 막고 성능을 최적화하는 데 사용된다.
- 웹 서버 미션에서의 '좋아요 버튼'에도 적용해볼 수 있을 것 같은데, 이는 좋아요 버튼의 값이 로그인 여부에 관계없이 클릭할 때마다 오르는 구조이기 때문이다.

# 4. 주간 프롬프트 회고
### 1. 내가 최종 선택한 가장 잘 작성했다고 생각하는 프롬프트
원본 프롬프트:
````text
DB를 인메모리에서 파일 기반으로 바꿀 때 인증 정보 관련 로직(SessionManager, AuthenticationContext/Filter/Manager 등)도 변경하는게 좋을까?

왜냐하면 이전에는 DB가 인메모리여서 DB와 SessionManager가 동일한 User의 참조값을 공유하고 있었기 때문에 User 객체를 수정하면 SessionStore에 저장된 User도 연달아 갱신이 되었지만, DB가 H2(파일)로 변경되는 순간 더 이상 참조값을 공유하지 않아서 발생하는 문제들이 있잖아.

변경한다면 SessionStore가 <SID, User> 대신 <SID, User.id or User.userId> 를 저장하는 방법도 생각해볼 수 있을 것 같은데, 기존 방법 대비해서 더 효과적일지에 대해 알려줘. 
````

선택한 이유:
1. 지난 프롬프트까지는 대화형 AI 기반이어서 컨텍스트(관련 코드)를 복사 붙여넣기로 직접 주입해 줬지만, 이번 프롬프트에서는 에이전트 환경인 점을 고려하여 컨텍스트로 전체 코드가 아닌 '관련 클래스명만 제시'하였다. 
2. 인메모리 DB에서 파일 기반 DB(H2)로 인프라 환경(기반)이 변경될 때 발생하는 객체 참조 및 상태 불일치(영속성 단절) 문제를 정확하게 인지하고 질문했다.

### 2. 원본 프롬프트 구조 분석
- **문제 상황:** 
  - DB가 인메모리에서 파일 기반(H2)으로 변경되면서, 기존 `SessionManager`와 DB가 `User` 객체의 동일한 참조값을 공유함으로써 얻던 상태 동기화 이점이 사라지고 정합성 문제가 발생할 수 있는 상황
- **원하는 결과를 어떻게 요구했는가:** 
  - 인증 정보 관련 로직의 변경 필요성을 묻고, `SessionStore`에 `User` 객체 대신 식별자(`User.id`)를 저장하는 대안이 기존 방식 대비 더 효과적일지 비교 분석해 달라고 요구했다.
- **필요한 맥락이 포함되어 있는가:** 
  - 이전 환경에서의 동작 원리(참조값 공유)와 변경된 환경(H2), 그리고 영향을 받는 핵심 클래스들(`SessionManager`, `AuthenticationContext` 등)이 명시되어 있다. 
  - 특히 에이전트 기반 환경임을 고려하여, 전체 코드 대신 관련 클래스명만 언급하였다.
- **판단 기준이나 제약 조건이 있는가:** 
  - 기존 방법(`<SID, User>`)과 새로운 대안(`<SID, User.id>`)을 구체적인 비교 대상으로 설정하여 답변의 범위를 한정했다.
- **출력 방식이 명확한가:** 
  - 특정 출력 형식(표, 마크다운 등)을 명시적으로 제한하지는 않았으나, 대안의 효과성에 대한 논리적인 비교 설명을 기대하고 있다.
- **나의 가설, 우려, 의도가 드러나는가:** 
  - DB 변경에 따른 데이터 정합성 문제에 대한 우려
  - 세션 저장소에 전체 객체 대신 식별자만 보관하여 의존성 및 결합도를 낮추는 것이 더 나은 설계일 것이라는 가설과 의도


### 3. AI에게 프롬프트를 개선 요청하기
- 프롬프트 개선안
```text
현재 프로젝트의 DB를 인메모리 환경에서 파일 기반(H2)으로 마이그레이션 중입니다. 
(에이전트는 현재 작업 공간의 SessionManager 및 Authentication 관련 로직들을 참고해 주세요.)

기존 인메모리 환경에서는 DB와 SessionManager가 User 객체의 동일한 참조값을 공유하여 객체 수정 시 상태가 자동 동기화되었으나, H2로 전환되면서 참조가 끊어져 정합성 문제가 발생할 것으로 예상됩니다.
        
이를 해결하기 위해 SessionStore의 저장 방식을 기존 `<SID, User>`에서 식별자 기반인 `<SID, User.id>` 형태로 변경하는 대안을 고려 중입니다. 

다음 두 가지 항목에 대해 텍스트로 분석 결과를 제공해 주세요.
1. 저장 방식을 변경했을 때와 유지했을 때의 장단점을 '데이터 정합성', '메모리 효율성', 'DB I/O 성능(조회 쿼리 증가 여부)' 측면에서 상세히 비교해 주세요.
2. 파악된 코드베이스 구조를 고려할 때, 제안한 대안(`<SID, User.id>`)을 적용하기 위해 수정이 필요한 핵심 클래스와 로직의 범위를 알려주세요.
```

### 4. AI가 개선한 프롬프트 분석
- **더 명확해진 목표 및 출력 형식:** 
  - 원본 프롬프트의 포괄적인 질문을 1번(다각적 장단점 비교)과 2번(실제 수정 범위 도출)으로 분리하여, 에이전트가 구조화되고 구체적인 형태의 답변을 생성하도록 유도했다.
- **추가된 제약 조건:** 
  - 에이전트 기반 환경의 이점을 살려 '현재 작업 공간의 로직을 참고하라'는 지시를 명시적으로 추가했다. 
  - 이를 통해 불필요한 코드 텍스트 주입을 방지하면서도 에이전트가 스스로 정확한 컨텍스트를 파악하도록 제한을 두었다.
- **구체적 판단 기준 제시:** 
  - 단순히 '기존 대비 더 효과적일지'를 묻던 것에서 벗어나, '데이터 정합성', '메모리 효율성', 'DB I/O 성능'이라는 3가지 명확한 엔지니어링 평가 기준을 설정했다.
- **답변 품질이 좋아질 것으로 예상되는 이유:** 
  - 아키텍처 설계 시 고려해야 할 트레이드오프 기준이 명확해졌기 때문에 모호한 답변을 피할 수 있다. 
  - 또한, 에이전트가 실제 코드베이스를 분석하여 수정 범위를 함께 제시하므로, 단순한 개념 비교를 넘어 실무에 즉시 적용 가능한 수준의 가이드를 도출할 수 있다.
