## 작업 과정에서 어려웠던 점 & 배운 점
### 정적 리소스와 템플릿 파일 분리
정적 리소스만으로 해결되었던 이전 단계들과 달리, 6단계에서는 화면에 로그인 유저의 닉네임을 출력해줘야 하는 요구사항이 있었다.

사용자별로 다른 화면을 보여줘야 했기 때문에, 더 이상 정적 리소스만으로는 미션을 진행할 수 없겠다고 생각했다.

그래서 템플릿 파일 개념을 도입하게 되었는데, 이 과정에서 크게 3가지의 서브 작업을 수행하게 되었다.

1. 프로그램에서 다루는 리소스의 종류가 늘어났기 때문에, 기존 `StaticResourceLoader`의 기능을 확장하고 이름을 `ResourceLoader`로 변경하였다.
2. 템플릿 파일은 `/resources/templates` 디렉터리에 모아뒀기 때문에, 정적 리소스와 속한 디렉터리가 달라지게 되었다. 기존에는 상대 경로를 사용하고 있었으나, 이는 리소스 서빙 안정성이 떨어지는 방식이라고 생각하여, 절대 경로로 수정하였다.
3. 템플릿 파일은 확장자가 포함된 경로로 접근하지 못하도록 제한하였다. 정적 리소스와 달리 템플릿 파일에는 템플릿 변수가 포함될 수 있어, 동적 렌더링 과정을 거치지 않으면 템플릿 변수가 그대로 노출될 수 있었기 때문이다.

단순히 서브 작업만 수행했으면 수월했겠지만, 그 과정에서 추가로 고려해야 할 문제들이 곳곳에서 등장하게 되었다. 가령,

1. 사용자별로 다른 화면을 보여줄 필요가 없는 로그인/회원가입 페이지는 정적 리소스로 제공할 수도 있었으나, 결과적으로는 템플릿 파일로 제공하게 되었다. 모든 파일에서 공통적으로 나타나는 헤더를 템플릿 변수 하나로 관리할 수 있는 템플릿 파일과는 달리, 정적 리소스는 공통 헤더가 변경될 경우 일일이 수정해줘야 하는 불편함이 있었기 때문이다.
2. 기존에는 단순 정적 리소스 요청을 `StaticResourceController`가 처리해줬지만, 템플릿 파일을 도입하고 보니 템플릿 파일 요청을 처리해주는 컨트롤러가 필요해졌다. 렌더링해야 할 내용들이 다를 경우 당연히 컨트롤러도 매번 생성하는 것이 맞겠지만, 내용이 동일한데 컨트롤러를 매번 생성해야 하는 구조에 대해서는 의문을 갖게 되었다. 하지만 이를 개선하는 작업은 리팩토링 과정이므로, 기능 추가를 목적으로 하고 있는 현재 커밋에서는 수행하지 않는 것이 맞겠다고 판단했다.

일련의 작업을 끝내고 보니 수정한 파일이 많아 ‘하나의 커밋으로 둬도 될지’에 대한 고민이 생겼는데, 결국 ‘동적 HTML 응답’이라는 하나의 기능 구현 과정에 필요한 작업들이었으므로 유지하는 것이 맞겠다고 생각하였다.

# 3. 학습 내용
## HTTP
### 멀티파트 데이터 
- 클라이언트가 서버로 여러 종류의 데이터를 동시에 업로드할 때 사용하는 HTTP 요청 방식이다.
- 각 데이터가 시작되고 끝나는 시점을 임의의 문자열(boundary)로 구분한다.

## JVM
### 동적 로딩
- 클래스 로더가 메모리(Method 영역)에 클래스 파일을 적재할 때 거치는 '로딩 → 링킹 → 초기화' 과정을 의미한다.
  - **로딩:** 클래스 파일을 읽어 메서드 영역에 클래스 정보(클래스/메서드/필드 시그니처, 런타임 상수 풀, static 변수 등)를 저장한다.
  - **링킹:** static 변수를 위한 메모리를 할당하고, 심볼릭 참조를 실제 메모리 주소로 교체한다.
  - **초기화:** static 변수에 실제 값을 할당한다.

### 런타임 데이터 영역
- 클래스로더가 클래스파일을 메모리(Method 영역)에 적재하기 위해, OS로부터 할당받는 메모리 영역 
- 데이터의 성격과 생명주기에 따라 5개로 나뉜다.
  - **힙 영역:** 프로그램 실행 중에 new 연산자로 생성된 객체와 배열
  - **메서드 영역:** 클래스/메서드/필드 시그니처, 런타임 상수 풀, 클래스 변수
  - **스택 영역:** 메서드 호출 시 생성되는 스택 프레임
  - **PC 레지스터:** 현재 실행 중인 JVM 명령어의 주소
  - **네이티브 메서드 스택:** 자바 외의 언어로 작성된 네이티브 코드를 실행하기 위한 공간
